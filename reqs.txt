La columna 'Implementación Técnica y Descripción' de cada tabla muestra en la parte superior el código y componentes involucrados, y en la parte inferior una

1. Arquitectura del Sistema (MVVM + Repository)
SmartPantry implementa el patrón MVVM complementado con el patrón Repository. La View solo dibuja; el ViewModel gestiona el estado y la lógica de pantalla;
el Repository es la única fuente de verdad para los datos.

02Persistencia 
Offline (SQLite)La app opera 
completamente sin 
conexión usando SQLite 
como base de datos local,
gestionada con sqflite o 
drift.InventoryRepository → sqflite 
query('products') → fallback a

El InventoryRepository implementa una 
estrategia cache-first: siempre lee 
primero de la base de datos SQLite local. 
Si se detecta conexión a internet 
mediante connectivity_plus, sincroniza los
cambios pendientes con Firestore. Esto 
garantiza que el usuario nunca vea una 
pantalla vacía por falta de conexión, y 
que la app funcione sin interrupciones en 
zonas con señal intermitente.Con modo avión activo, 
el inventario carga en ≤ 
500 ms desde SQLite.

03Sincronización 
FirebaseLos datos locales se 
sincronizan con Firestore 
al recuperar conexión, sin 
pérdida de información.connectivity_plus → 
syncLocalToFirestore() → batch 
write → synced=true en SQLite
El InventoryRepository escucha el Stream
de conectividad de connectivity_plus. 
Cuando detecta que la conexión vuelve, 
ejecuta syncLocalToFirestore(), que 
consulta los registros de SQLite con 
synced=false y los escribe en Firestore 
usando un batch write (escritura en lote) 
para optimizar las operaciones de red. Al 
confirmar el éxito, marca esos registros 
como synced=true en SQLite para evitar 
duplicados.Cambios realizados

4.1 Flujo: Escanear y Agregar un Producto
ScannerView → ScannerViewModel → ApiService (OpenFoodFacts) → InventoryRepository → SQLite + Firestore
# Paso del Flujo Técnico
1 View: ScannerView activa MobileScanner widget → cámara del dispositivo inicia (requiere permiso camera en 
AndroidManifest e Info.plist).
2 View → ViewModel: onDetect(barcode) dispara ScannerViewModel.onBarcodeDetected(barcode: String). Se 
aplica throttle para evitar múltiples llamadas.
3 ViewModel → Service: ScannerViewModel llama ApiService.getProductByBarcode(barcode). El servicio ejecuta 
GET https://world.openfoodfacts.org/api/v0/product/{barcode}.json con http.get() envuelto en try-catch.
4 Service → ViewModel: Si status == 1 → parsea product_name, image_url y retorna ProductModel parcial. Si 
status == 0 → lanza ProductNotFoundException → ViewModel activa modo ingreso manual.
5 ViewModel → View: ViewModel expone foundProduct (ProductModel?) y uiState (loading / found / notFound). La 
View reacciona con Consumer<ScannerViewModel>.
6 View → ViewModel: Usuario selecciona fecha rápida (+3d, +1sem) o abre showDatePicker() nativo → 
ScannerViewModel.setExpiryDate(DateTime).
7 ViewModel → Repository: Usuario pulsa 'Add to Pantry' → ScannerViewModel.confirmAdd() → 
InventoryRepository.addProduct(product) → guarda en SQLite y Firestore.
8 Repository → View: addProduct() completa → ViewModel emite ProductAddedState → ScannerView navega de 
vuelta con Navigator.pop() y muestra toast de confirmación.
4.2 Flujo: Cargar Recetas por Ingredientes Próximos a Vencer
RecipesView → RecipesViewModel → InventoryRepository (SQLite) → ApiService (TheMealDB) → RecipesView
# Paso del Flujo Técnico
1 View: RecipesView se inicializa → ConsumerWidget llama RecipesViewModel.loadRecipes() en initState.
2 ViewModel → Repository: RecipesViewModel llama InventoryRepository.getExpiringProducts(withinDays: 3) → 
SQLite query filtra productos con expiryDate <= hoy + 3 días.
3 ViewModel → Service: Por cada ingrediente expirante, ApiService.getRecipesByIngredient(ingredientName) → 
peticiones paralelas con Future.wait([...]).
